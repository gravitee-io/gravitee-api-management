# V4 Analytics Dashboard – Test Data Profile

This document describes the data generated by `scripts/generate-test-data.sh` and what you should expect to see in each dashboard widget when viewing the **analytics-test-api** in the Console (API Traffic → Analytics).

## How the data is generated

The script:

1. **Creates or reuses** a V4 HTTP Proxy API named **analytics-test-api** (context path `/analytics-test`), backed by **https://httpbin.org**.
2. **Deploys** the API so the gateway proxies requests to httpbin.
3. **Sends 100+ requests** over a short period (a few seconds, with ~0.8 s pauses between batches) to produce:
   - **Mixed HTTP methods:** GET, POST, PUT, DELETE.
   - **Mixed status codes:** 200, 201, 400, 404, 500, 503 via httpbin paths:
     - `/get`, `/post`, `/put`, `/delete` → typically **200** (or 201 for POST).
     - `/status/200`, `/status/201` → **200**, **201**.
     - `/status/400`, `/status/404`, `/status/500`, `/status/503` → **400**, **404**, **500**, **503**.
   - **Mixed request sizes:** small JSON bodies and one larger body per batch (~2KB) to vary `request-content-length` and latency.
4. **Waits 15 seconds** for Elasticsearch bulk indexing, then **verifies** `gravitee-v4-metrics-*` document count.

Traffic is sent to the **gateway** (e.g. `http://localhost:8082/analytics-test/...`). Metrics are stored in the **v4-metrics** Elasticsearch index and exposed by the unified analytics endpoint and existing line-chart endpoints.

## What to expect in each dashboard widget

Use a timeframe that includes the run (e.g. **Last 24 hours** or **Last 7 days**). After running the script and waiting for indexing, refresh the dashboard.

### 1. Request Stats (four cards)

- **Total Requests (COUNT)**  
  Should show **100+** (or the total from the script summary). Same value as the number of requests sent in the selected period.

- **Avg Gateway Response Time (STATS on `gateway-response-time-ms`)**  
  A positive average (e.g. tens to low hundreds of ms). Varies with network to httpbin.org and gateway load.

- **Avg Upstream Response Time (STATS on `endpoint-response-time-ms`)**  
  Similar order of magnitude to gateway time; reflects httpbin response time.

- **Avg Content Length (STATS on `request-content-length`)**  
  Mixed small and larger bodies → average somewhere in the hundreds of bytes (script uses small payloads and one ~2KB body per batch).

### 2. HTTP Status (GROUP_BY on `status`)

- **Pie (or bar) chart** with segments for **200**, **201**, **400**, **404**, **500**, **503**.
- **200** should be the largest segment; **201** and 4xx/5xx smaller and similar to each other (each ~5 per run).

### 3. Response status over time (line chart)

- **Existing endpoint:** `/analytics/response-status-overtime`.
- Multiple series (e.g. 2xx, 4xx, 5xx) over time.
- You should see a **short spike** in the time window when the script ran (requests concentrated over a few seconds), with 2xx dominating and smaller 4xx/5xx contributions.

### 4. Response time over time (line chart)

- **Existing endpoint:** `/analytics/response-time-over-time`.
- Single series (e.g. average response time per bucket).
- A **short spike or plateau** in the same time window, with values in the tens–hundreds of ms range depending on httpbin and network.

## Timeframe and indexing delay

- **Predefined ranges** (Last 5 min, 1h, 24h, 7d, 30d) use “now” in UTC; ensure the script ran within the selected range.
- Allow **~15–30 seconds** after the script finishes before expecting all points in the dashboard; the script already waits 15 s and then checks `gravitee-v4-metrics-*/_count`.
- If the dashboard shows **no data**, try “Last 24 hours” or “Last 7 days” and refresh; confirm Elasticsearch has documents for this API in `gravitee-v4-metrics-*`.

---

## Re-running the script (idempotent)

- The script is **idempotent**: it looks up an API named **analytics-test-api** in the same environment. If found, it reuses that API and **redeploys** it; if not, it creates a new one.
- Each run **adds** more requests. Totals and distributions in the dashboard will reflect **all** traffic in the selected timeframe (including previous runs).
- To start from a clean slate, delete the **analytics-test-api** in the Console (or via Management API) and run the script again; it will create a new API.
